Preamble

This document serves as a general style guide in order to keep the project's formatting consistent and to resolve any organizational disputes before they happen by proposing a set of guidelines to follow. These will cover all sorts of subjects, from case to naming to structure. Some of these will be more opinionated than others, or might warrant a justification: in general, a short reason for the style will be given, as well as a description of what it entails. For the sake of familiarity, the code examples will be given in Java, unless otherwise noted. It's also worth noting that a lot of these rules were written with C-style languages in mind. This guide might thus be completely ridiculous, or simply incompatible, with other languages whose syntax may greatly differ.

This document is currently a draft; please feel free to suggest any changes to it by linking the relevant suggestion.

Spacing

"Spacing" generally refers to the space between characters in source code, whether it be indentation or spacing on a particular line. In languages inheriting their syntax from C (appropriately-named "C-style" languages), whitespace is purely cosmetic; for this reason, various opinions have been formed over where it should and shouldn't go, and how much there should be. Here's ours.

Note: spacing rules, when defined as "before this" or "after that", don't stack: for example, as will be discussed, the plus operator should be surrounded by spaces, and the unary negation operator should be preceded by one. However, if an unary subtraction follows an addition, there should only be one space between the two—write 1 + -x. In a similar vein, if an operator states that no space should follow it, but the operator following it states that it should be preceded by one. For example, as will be seen, closing angle brackets are not followed by spaces, but opening curly braces are—the correct resolution is thus Array<T> {}.

Indentation

Indentation is done with spaces, not tabs, and the spaces per level is 4. Blocks always add an indentation level, and if an expression is broken onto multiple lines, a level is also added.

The exception to this rule is the indentation of documentation comments: the opening segment (the /**) is indented normally, but every line thereafter, including the closing segment (the */), is indented with a single additional space.

Around operators

Generally, most operators are surrounded by spaces: the arithmetic, comparison, relational, bitwise, logical and assignment operators all take spaces before and after them. However, unary operators only take one space, and its position depends on if the operator is postfix or prefix: in the case of a prefix operator, the space goes before the operator, and in the case of a postfix one, it goes after, unless a punctuation mark or closing bracket is present, in which case no space is placed. This rule takes precedence over the previous, so even though the ! (boolean negation) operator is a logical operator, because it is an unary operator, no space should be inserted after it. Notably, the comma operator (,) is treated as punctuation for the purposes of spacing rules.

Note that depending on the language, a symbol can mean different things: Perl uses the dot (.) as a string concatenation operator—because its role is more similar to addition than it is to property access, it should follow the rules for the addition operator, namely, a space before and after.

Around punctuation, brackets and other symbols

Punctuation—such as commas (,), semicolons (;) and colons (:)—generally take no space before them and a space after them, unless they are the last character on their line, in which case they take no space.
It's worth noting that the colon is somewhat troublesome in that it sometimes takes no space before and sometimes does. The user's common sense will take over, but some cases worth noting are as follows—the colon should be surrounded by spaces when used in places similar to Java/C++'s 'for-each' loops (for (ele : array)), and when used as the 'else' clause of a ternary operator. It should inversely not be preceded by a space when used with a label or when used with a switch statement's case expression.

Brackets fall into four types: parentheses (()), square brackets ([]), curly braces ({}) and angle brackets (<>)—each have different uses and thus, different rules.

Parentheses are used mostly for denoting order of operations, like in mathematics, for performing function calls and declaring functions. In the first case, the parenthetical unit should be surrounded by spaces, but the content of the parentheses should not be padded by spaces: write 2 * (1 + x) + 3 and not 2 * ( 1 + x ) + 3 or 2 *(1+x)+ 3. In the other two cases, place no space before the parentheses, and do not pad its contents.

Square brackets are primarily used to index arrays, declare array types and create them. In all three cases, no spaces should pad the brackets' internal content, and no space should surrounded the brackets—write int[] x = array[2]; and not int [] x = array [ 2 ];.

Curly braces are extremely important, because they section off parts of code—they're boundaries. A defining trait of a C-style language is its use of curly braces for this role. Usually following a control flow statement such as an if, for or try, they should have a space before them and not a line break. As stated in the indentation section, the creation of a block for any reason increases the indentation level within that block by 1. If both curly braces are on the same line, however, there is obviously no indentation within the block. In that case, the curly braces' contents should be padded by spaces, unless there are no contents to be padded, in which case there should be no spaces between the curly braces—write { something(); } and if (true) {}, not {something();} and if (true) { }.

Angle brackets, when not used as comparison operators, are usually used in conjunction with generics, either declaring them or specifying them. In both cases, they take no spaces before, after nor inside.

When writing comments, put a space after the start of the comment (a double slash in C-style languages //, sometimes a hashtag # in languages like Python) for legibility: do the same regardless of the type of comment (line/block).

Blank lines

Blank lines should be used to break up large segments of code, or just generally to break up pieces of code that isn't related to each other. For example, it's common to declare a host of variables at the top of a function, and then perform operations on those and according to those variables. Since these are two different sections of the functions, a blank line between them can help the reader to parse each section. Never use more than one blank line at a time. Here are some common places where blank lines are used to break code up. (Note that blank lines at the very start/end of a file are pointless, and so if, for example, a class spans the entirety of a file, it should not be padded by blank lines.)

  • Before and after anything that creates a block (if statement; for loop; function, class, enum, interface or other similar declaration, user-defined blocks)
  • Between a class' attributes and its constructor(s)
  • Between a class' methods
  • Before and after a block comment
  • After long method chains or otherwise "heavy" code (repeated concatenation, repeated arithmetic operations, etc)
  • After a group of variable declarations, unless a return immediately follows and the function ends there.

Naming conventions

The term "naming convention" can cover a variety of things, but they're all in some way related to the way in which a program's variables are named. Naming conventions matter, because a proper one achieves consistency within the codebase, and can help add clarity to it.

Naming case

This refers to how variables with names encompassing multiple words should be written: the most common (for C-style languages) are as follows.

Common variable cases
Name Example
Camel Case someVariable
Pascal Case
Upper Camel Case		 SomeVariable
Snake Case some_variable
Screaming Snake Case SOME_VARIABLE

Default to using camelCase for naming symbols, and use the following cases under the following conditions.

  • Classes, enums, interfaces, records, type aliases, structs, decorators, generics and similar constructs should use PascalCase. Do not prefix interface names with "I".
  • Class attributes marked as static and readonly use SCREAMING_SNAKE_CASE. Note that this does not apply to enum members, record members or other class attributes. In languages where interface members translate to static members on implementing classes (such as Java), interface members marked as readonly should use SCREAMING_SNAKE_CASE.
    • Notably, local variables and instance attributes never use SCREAMING_SNAKE_CASE, even if marked as readonly.
  • In the presence of acronyms, capitalize the acronym's first letter if it is not the first word in a camelCase name and leave everything else lowercase. If it is anywhere in a PascalCase name, the whole acronym is in uppercase—write XMLHTTPRequest xmlHttpRequest.
  • Type aliases that alias a type whose name doesn't start with a capital letter (such as an alias for int) may also be written without a lowercase if desired—the type alias thus uses camelCase instead of PascalCase.

Variable names

Looking more specifically at what to name the symbols in the program, there are a couple conventions to be established in order to help inference and general clarity.
Variable names should be succinct, but clearly describe the value they represent. There is a balance to be struck between clarity and conciseness: i is not a good variable name, because it conveys no information; inversely, integerStoringTheHealthOfThePlayer is way too long, and can be shortened to just health, or playerHealth—the surrounding context should be enough to fill in the blanks as to whose health is concerned. This is important: the amount of information in a variable's name depends on how much information is given by the surrounding context. They should also only be comprised of letters and numbers.

For example of this would be functions describing mathematical formulae: in this context, the parameter has no real meaning beyond being an input to a simple mathematical function. Moreover, most people will be familiar with a one-letter variable name, as mathematical variables tend to be one-letter long. It is thus totally acceptable, even recommended, to have a function signature resembling double sin(double x), or double atan2(double y, double x).

Another example would be a function to move an object to some location: if the function is called something like moveTo, it's completely fine to use x and y as parameter names, because there's no ambiguity as to what they mean—the signature thus resembles void moveTo(double x, double y). In addition, one-letter variable names are usually used in places such as for loops, because the surrounding context—the loop—provides enough context to know what i represents.

Only use abbreviations or shortened forms if they are significantly shorter than the full form and it is easy to figure out the lengthened form. For example, things like "HTTP", "XML" or "ms" are encouraged, because the full names would be unwieldy ("HyperTextTransferProtocol", "eXtensibleMarkupLanguage" and "millisecond") and because it's easy to know what is meant Inversely, abbreviations like "val" for "value", or "str" for "string", are unnecessary and somewhat lazy.
Single-letter identifiers for generic types are usually fine, it usually depends on the context, and so good judgement should do the trick.
And for the love of all that is sacred, do not prefix any variable name with "my"!!! Not only is the variable not "yours", even if it were, no one would care.

Function names

Functions do things, and thus, their names should describe the things they do. The description of an action usually involves a verb, and thus, function names should start with a verb, either conjugated or infinitive. Like variable names, the amount of detail regarding the description of the action the function performs should depend on the context it finds itself in.

Some functions return a boolean value communicating the state of some thing or things—for example, if a collection is empty, or if it is sorted; if two objects are equal or compatible. These functions are often called "predicates". The type can further be narrowed based on the number of arguments, which usually dictates the predicate's role, but all of them start with a verb, and it is usually conjugated to a present tense.

  • Predicates with no parameters report the state of the object they are attached to. For example, the isEmpty method of the String class takes no arguments, and reports on the state of the object it's attached to—specifically, on whether or not it is empty.
    These predicates' names should start with the third person singular form of a verb. Usually, the verb "to be" is conjugated, giving "is"—isEmpty, isSorted, isReady—but other verbs, like "to have" and "to be able to", can be used—hasNext, hasKey, canBeSerialized.
    Alternatively, an adjective qualifying the state of the object can be used—writable, enumerable, closable.
  • Predicates taking one parameter can do one of two things: if they compare the object they receive to the one are attached to—such as Java's equals method—they use the third person singular present form of a verb, as they establish a relationship between a "current" object and another—equals, contains, hasKey. Sometimes, there is no verb that expresses the relationship the predicate is trying to express: in this case, go back to using "is", followed by whatever the predicate asserts—isSubstringOf, isOwnedBy.
    In the case of predicates that take a single parameter and report some state of that object, they behave like no-parameter predicates, except that they report the state of the object they're given, rather than the one they're attached to. They therefore follow the same rules as no-parameter predicates.
  • Predicates taking more than one parameter establish a relationship between multiple objects, which may or may not include the one the predicate is attached to. In the case where the objects are compared amongst themselves without the predicate's owner, the predicate's name should begin with the third person plural form of a verb—areEqual, areCompatible.
    If the owner's object is involved, the predicate will usually be asserting a one-to-many relationship; thus, the verb should be third person singular—owns, canCall, hasControlOver. Note that these predicate names could also be used for one-parameter predicates—if extra clarity is needed, the word "all" can be appended—canCallAll, hasControlOverAll.
It's possible for a predicate to have a verb conjugated to a past or future tense—wasReady, willRead—although this is rare. Boolean attributes, being more or less like no-parameter predicates, should follow the rules governing them.
